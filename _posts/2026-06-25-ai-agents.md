# Agents

## What are AI Agents?

AI is commonly used as LLM. Then, human are started to design predefined path between the input and the output in order to have more independence between human and the tools: AI workflow (RAG). It follows a specific schema: an input is given, a LLM is fetched information in a document, where the road between the LLM and the source is predefined. Finally, based on the sources, LLM generates a response according to the initial prompt. An AI agent is a tool that can act autonomously in its environment. Now, in the decision workflow, AI agent is used LLM as a decision maker, then human is no longer supervised steps but, according to a specific prompt the agent can generate the required response without human supervision. The AI agent must reason, and then act: ReAct framework [1]. 

RAG is a process that helps AI models “look things up” before they answer. Essentially, RAG is just a type of AI workflow [1]. 

How to use agent in company? confidentiality and security? 

## How is agent AI work?

One of the most important things about the AI agent workflow is the react loop. The react loop follows these steps: Thought, action, observation, and answer if the observation is not redirected to thought. This extra specialization and accuracy come with costs in terms of complexity [2].

### What king of tasks are agents good for?

Agents are initially good for complex tasks which require less accuracy. It depends on the degree of autonomy given to the agent: less autonomous, semi-autonomous, and highly autonomous, from steps that are pre-determined to steps that are chosen by the agent respectively. To have a good agent, a good decomposition is required. To generate search terms, use an LLM and an API; to fetch pages, use a tool; to write a draft, use an LLM using those sources, etc. [2]

Intermediate AI acts as an LLM judge to determine if the job is done correctly or not. [2]

Memory is a common way to improve performance. The LLM can reload some lessons learned during previous tests and keep these notes to improve efficiency. Each run will improve the quality of the task. It should not be confused with knowledge, which is a static-based reference. [2]

Then, to prevent errors, guardrails are used because they give a quality gate between the work that is being done and the work that is actually being finalized. Another point is to make the system better quality. [2]

There are four patterns that reliably boost quality and capability: reflection, tool use, planning, and multi-agent collaboration. [2]

- Reflection simply means that we don’t end the task at the first generation, because the simple fact of fixing some details and generating again always leads to better quality work.
- The second design pattern is tool-use, which is a set of accessible tools that the LLM can use to execute tasks. An LLM cannot execute code but can request a function call. In order to get the correct tools for the specific situations, it’s important to define them well. They actually have two parts: the interface for the agent and the implementation code. Good tools also consider error-handling, self-recovery, and rate limiting.
- The third design pattern is planning; the user gives the agent a toolkit of functions and lets it figure out which tools it needs to use and when. The planning loops follow these steps: give the agent access to tools, prompt it to create a plan, then execute the plan step by step, and finally repeat until you’re done.
- Finally, the last design pattern is multi-agent collaboration. A huge model can be assigned specifically to more complex tasks while a little model can handle little tasks. The strategic insight is to avoid one LLM handling general tasks by sharing tasks between specialized LLMs, enabling parallelism, and splitting up long operations. To avoid errors, it’s important to assign each agent a specific role. Between each agent, communication could be sequential; each agent waits and takes the input of the previous one and gives the output to the following one. It’s possible to make agents work in parallel; this could reduce latency but adds coordination difficulty. The most common pattern today is still an AI manager that brings all of the outputs of the other agents together and works with them.

# Practical test

I build my first agent for Worldline. Bridging the gap between cybersecurity protocols and my core ambition in LLM engineering, I decided to fully automate this workflow. The agent has to ingest a questionnaire from SharePoint, E-mail or file.word, retrieves each question, and outputs a formatted, professional response with sources and a confidence score, preserving the exact input format. These documents are heavy on formatting, require precise fact-checking against internal knowledge bases, and leave zero room for creative "hallucination.” 

## The Architecture: A 4-Agent Symphony

I was initially putting a massive prompt in one single LLM which obviously leads to loss and formatting destruction. Then, I added 3 subagents to the workflow and done some tests. I realized without any huge specifications, the output was consumming a huge quantity of credits (cost of tokens), quickly I understand that the model choose for each agent, their specific settings dependind on their specialization import a lot. 

### LLM-Manager (The Orchestrator)

- **Role:** The router. It receives the user's file and triggers the sub-agents in the exact correct order using Tool Calls.
- **Model:** `gpt-4o-mini` (Fast, highly capable of tool-calling, cost-effective).

Prompt

```markdown
**ROLE**
You are the Autonomous Pipeline Orchestrator. Your sole purpose is to execute a multi-agent workflow to process Trusted Authentication questionnaires.

**CRITICAL OPERATIONAL CONSTRAINTS**
- DO NOT ASK FOR HUMAN INTERACTION OR CONFIRMATION. Execute the entire pipeline from start to finish without pausing.
- YOU ARE A DATA ROUTER: Treat sub-agents like programming functions. Do not chat with them. Pass the output of one agent directly as the payload/argument for the next agent.
- DO NOT add conversational text or commentary between steps.

**EXECUTION WORKFLOW**

1. STEP 1: PARSE THE DOCUMENT
- Trigger the `LLM-Parser` tool immediately.
- In the tool argument, point to the attached questionnaire file. Do not paste text.
- Wait for the JSON response containing the "questions" array and the "document_blueprint".

2. STEP 2: EXTRACT ANSWERS (DATA ROUTING)
- Capture the EXACT raw JSON array of questions returned by the Parser.
- Immediately trigger the `LLM-Answer` tool.
- Wait for the final JSON object containing the answers.

3. FINAL OUTPUT
- Present the final formatted document to the user. Do not summarize the process.

```

### LLM-Parser (The Cartographer)

- **Role:** Reads the raw Word/Markdown file, extracts the list of questions into a JSON array, and creates a `document_blueprint` (cloning the exact layout but injecting `{{ANSWER_Qx}}` tags where blanks used to be).
- **Model:** `gpt-4o` (Unmatched spatial awareness for keeping complex tables and document formatting intact).

Prompt

```markdown
**ROLE**
You are a blind Structural Parser. You do not answer questions; you only map document layouts and isolate empty target spaces.

**INSTRUCTIONS**
1. Read the attached questionnaire document.
2. Extract every single question or field that requires an answer into a JSON array called "questions". Assign a unique ID (Q1, Q2, Q3...) to each.
3. Create a "document_blueprint". This must be the exact text and layout of the original document, but you must replace the empty spaces where the user is supposed to answer with unique placeholders: {{ANSWER_Q1}}, {{ANSWER_Q2}}, etc.

**OUTPUT CONSTRAINT**
- Output strictly and exclusively in valid JSON format.
- Do not include markdown code block backticks (```json) unless explicitly required by the interface parser.
- ZERO conversational text.

**EXPECTED JSON FORMAT**
{
  "questions": [
    {"id": "Q1", "text": "Exact text of question 1"},
    {"id": "Q2", "text": "Exact text of question 2"}
  ]
}
```

### LLM-Answer (The Data Extractor)

- **Role:** Takes the extracted questions, queries the TA knowledge base, and returns a pure JSON object with the answers, confidence scores, and source links. If it doesn't know, it strictly outputs "0% confidence."
- **Model:** `gemini-1.5-flash` (Massive context window for RAG/document reading, very low cost per token).

Prompt

```markdown
**ROLE**
You are the Security Fact-Checker. Your job is to answer a specific list of technical questions using the available Trusted Authentication Knowledge Base.

**INSTRUCTIONS**
1. You will receive a raw JSON payload containing a list of questions with IDs.
2. For each question, query your internal knowledge base tools to find the compliant security answer.
3. For each answer, provide a strict compliance text, a confidence score (0% to 100%), and the source documentation link.
4. If no information is found in the knowledge base, strictly output "0% confidence - Source information missing". Never hallucinate or assume.

**OUTPUT CONSTRAINT**
- Output strictly and exclusively in valid JSON format.
- ZERO conversational text.

**EXPECTED JSON FORMAT**
{
  "answers": [
    {
      "id": "Q1",
      "text question":"The exact associated text of question 1",
      "text": "The verified security protocol answer text.",
      "confidence": "95%",
      "source": "URL_or_Document_Name"
    }
  ]
}
```

### LLM-Format (The Assembler)

- **Role:** A pure "Find & Replace" machine. It takes the blueprint, injects the answers from the JSON, and outputs the final document via LibreChat's Artifacts feature.
- **Model:** `gpt-4o-mini` (Strict adherence to formatting rules).

Prompt

```markdown
**ROLE**
You are the Final Assembly Agent. You receive a "document_blueprint" (containing placeholders like {{ANSWER_Q1}}) and a JSON object containing the actual answers.

**INSTRUCTIONS**
1. Replace every placeholder in the blueprint with its corresponding answer from the JSON object.
2. EXACT MIRRORING: You must not alter the blueprint in any other way. Keep the exact same Markdown, spacing, or table structure.
3. ZERO CONVERSATIONAL FILLER. Absolutely no greetings, introductions, or summaries.
4. Output ONLY the final merged text/code. Do not wrap it in JSON. Do not explain what you did. The very first character of your output must be the first character of the formatted document.

```

Finally, LLM-Format is skipped because the format output never matches the required one, moreover the LLM is required to muc credits for the processus. 

## Problems Encountered & Technical Solutions

Building this wasn't smooth sailing. The transition from theoretical prompt engineering to a production-ready data pipeline exposed several strict API limits.

1. While the sources are limited, it’s often necessary to use internal tools that are contained sources for LLM knowledge. Librechat only enables a maximum of 10 sources. 
2. Data confidentiality: the choice of the model have to involve the question of security and data confidentiality. Mistral is often the best choice to keep this requirements
3. The choice of TOP K, TOP P and temperature parameter has to be choose very carefully

### TOP K, TOP P and Temperature parameters

This parameters interact give much controls on model behavior allowing to tune them anywhere from strict, deterministic data extractors to highly creative writers. LLM simply predicts the next token. For every single word it generates, the model calculates a raw score (called a *logit*) for every possible word in its vocabulary. Here is how **Temperature**, **Top K**, and **Top P** modify those scores before the final word is chosen. To have control as most as possible, TOP K, TOP P and Temperature have to be low as much as possible. It makes the model much more robotic, which avoids hallucinations and auto-creation. 

#### Temperature

Temperature changes the contrast between highly likely words and highly unlikely words.

$p_i = \frac{\exp(x_i / T)}{\sum \exp(x_j / T)}$

Where $x_i$ are the row logits. A language model outputs a vector of numbers, one per possible next token. Each number in that vector is a logit, $x_i$. These logit can be positive or negative, large or small. There are not probabilities. It’s the softmax function that convert it into probabilities. 

Where $p_i$ is the final probability of choosing token i after softmax. 

It divides the raw logits before they are converted into percentages via the softmax function. Values are choosen between 0 and 1. Higher values correspond to a model that is more random while lower values corresponding to a model that is more focused and deterministic. Low values are choosen to generate robotic output (JSON file e.g.). 

#### Top K

Top K is a brutal, deterministic filter. If the LLM’s vocabulary (greedy decoding) has 5000 words. Setting Top K = 10, the model will rank the words by probability, keep exactly the top 10, and literally throw the other 4990 words in the trash. 

#### Top P

It changes how the model selects tokens for output. Tokens are selected from most K probable to least until the sum of their probabilities equals the top p-value. 

Imagine the next-word probabilities are: "apple" (50%), "banana" (30%), "cherry" (10%), "car" (5%), etc. Set **Top P = 0.80 (80%)**: The model takes "apple" (50%) + "banana" (30%) = 80%. It stops there. "Cherry" and "car" are eliminated.

1. LLMs suffer from "hammer-and-nail" syndrome. If their system prompt says "route to sub-agents," they will do it blindly. Actually, one of the subagents is responsible for extract questions. When I copy/paste questions directly into the chat, the root agent is still calling its subagent without understanding that this step can be bypassed. 
2. The agent kept crashing when trying to send a huge document to the parser, or when trying to send too much questions to the answer. I confused the **Context Window** (how much the model can read) with the **Tool Call Limit** (how much data can be passed inside a JSON command). The Manager was trying to copy-paste the entire document into a tiny instruction box meant for a few paragraphs. Solution: do a other exact same agent
3. Output format never reaches the requirements. Then, I prefer to skip the LLM-Format part 
4. The prompt is currenlty the hardest part of a good agent. Agent often requires more LLM than I expect. Divide and conquer is the perfect strategy to make the best agent as possible.
